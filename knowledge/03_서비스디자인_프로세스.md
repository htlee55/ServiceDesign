# 03_서비스디자인_프로세스

## 1. 문서 목적

이 문서는 AI 기반 안전·안심 서비스를 설계할 때 적용할 서비스디자인 프로세스를 정리한 지식자료이다.

나만의 GPT는 이 문서를 참고하여 사용자가 제시한 안전 문제를 단순 기술 적용 과제로 다루지 않고, 사용자·현장·이해관계자·운영 구조·비즈니스 모델·윤리 이슈를 포함한 서비스 설계 과제로 다루어야 한다.

서비스디자인 프로세스의 핵심은 다음과 같다.

1. 문제를 정확히 정의한다.
2. 사용자를 구체적으로 이해한다.
3. 위험상태 전이를 분석한다.
4. 이해관계자의 역할과 책임을 구분한다.
5. HMW 질문으로 설계 방향을 도출한다.
6. 서비스 컨셉을 만든다.
7. 사용자 여정과 폐루프 구조를 설계한다.
8. AI·데이터·운영 구조를 정의한다.
9. PRD 또는 서비스 기획서로 구체화한다.
10. 안전성과 안심 형성 가능성을 평가한다.

---

## 2. 전체 프로세스 개요

AI 기반 안전·안심 서비스 설계는 다음 순서로 진행한다.

```text
1. 문제 이해
→ 2. 문제 재정의
→ 3. 이해관계자 분석
→ 4. 사용자 페르소나 작성
→ 5. 위험상태 전이 분석
→ 6. 사용자 여정지도 작성
→ 7. Pain Point와 Opportunity 도출
→ 8. HMW 질문 도출
→ 9. 서비스 컨셉 도출
→ 10. 폐루프 서비스 구조 설계
→ 11. AI·데이터 구조 설계
→ 12. 안전층과 안심층 설계
→ 13. 운영모델과 비즈니스 모델 설계
→ 14. PRD 작성
→ 15. 평가와 개선
```

이 프로세스는 반드시 한 번에 모두 수행할 필요는 없다.
사용자의 요청에 따라 일부 단계만 수행할 수 있다.

예를 들어 사용자가 “문제정의부터 HMW까지”라고 요청하면 1~8단계까지만 수행한다.
사용자가 “PRD로 정리해 달라”고 요청하면 9~14단계를 중심으로 작성한다.

## 3. 1단계: 문제 이해

### 목적

사용자가 제시한 문제를 그대로 받아들이기보다, 현장의 맥락과 현재 대응 방식, 한계를 파악한다.

### 확인할 질문

* 어떤 현장인가?
* 누가 위험에 노출되어 있는가?
* 어떤 위험이 발생할 수 있는가?
* 현재는 어떻게 대응하고 있는가?
* 현재 방식의 한계는 무엇인가?
* 사고가 발생하면 누가 피해를 입는가?
* 누가 비용을 부담하는가?
* 누가 조치할 수 있는가?

### 산출물

* 현장 개요
* 핵심 사용자
* 위험상황
* 현재 대응 방식
* 현재 대응의 한계
* 설계해야 할 핵심 문제

---

## 4. 2단계: 문제 재정의

### 목적

표면적인 문제를 안전·안심 서비스 관점에서 다시 정의한다.

사용자가 제시하는 문제는 대개 다음과 같이 표현된다.

* CCTV가 부족하다.
* 경비원이 부족하다.
* 낙상을 감지해야 한다.
* 작업자가 위험행동을 한다.
* 보호자가 불안하다.
* 사고가 나면 대응이 늦다.

이 문제를 서비스디자인 관점에서는 다음과 같이 재정의한다.

* 사고 전 위험상태로 전이되는 과정을 조기에 발견하지 못한다.
* 감지된 위험이 실제 조치와 결과 확인으로 이어지지 않는다.
* 사용자가 위험이 관리되고 있다고 느끼지 못한다.
* 보호대상자와 관리자의 안전 역량을 증강하지 못한다.
* 현재 시스템이 안전은 일부 제공하지만 안심을 형성하지 못한다.

### 문제정의 문장 구조

```text
현재 [사용자/현장]은 [위험상황]에 노출되어 있다.
기존 대응은 [현재 방식]에 의존하고 있으나,
[한계] 때문에 사고 전 위험 전이를 조기에 발견하거나 실제 안전 여부를 확인하기 어렵다.

따라서 본 서비스의 핵심 문제는
[사용자]가 [위험상황]으로 전이되기 전에 이를 감지하고,
적절한 사람이 조치하며,
그 결과를 확인하여 사용자가 안전하다고 느낄 수 있게 하는 것이다.
```

## 5. 3단계: 이해관계자 분석

### 목적

안전·안심 서비스는 여러 사람이 함께 작동하는 서비스이다.
따라서 누가 위험에 노출되어 있고, 누가 조치하며, 누가 비용을 부담하는지 구분해야 한다.

### 이해관계자 유형

| 유형     | 설명                            |
| ------ | ----------------------------- |
| 보호대상자  | 실제 위험에 노출되는 사람                |
| 핵심 사용자 | 서비스를 직접 사용하는 사람               |
| 조치자    | 위험 발생 시 실제 조치를 수행하는 사람        |
| 관리자    | 상황을 모니터링하고 자원을 배분하는 사람        |
| 비용부담자  | 서비스 비용을 지불하는 사람 또는 조직         |
| 책임자    | 사고 발생 시 책임을 지는 사람 또는 조직       |
| 간접 사용자 | 가족, 동료, 입주민, 방문자 등            |
| 규제기관   | 법, 규정, 개인정보, 안전 기준과 관련된 외부 주체 |

### 산출물

| 이해관계자 | 현재 역할 | 불편/위험 | 서비스에서 기대하는 가치 | 조치 책임 |
| ----- | ----- | ----- | ------------- | ----- |

## 6. 4단계: 사용자 페르소나 작성

### 목적

서비스가 누구를 위해 설계되는지 구체화한다.

안전·안심 서비스의 페르소나는 단순 인구통계 정보보다 다음을 중시해야 한다.

* 위험에 노출되는 상황
* 현재 느끼는 불안
* 기술에 대한 수용성
* 감시로 느낄 가능성
* 실제 조치 능력
* 서비스로 얻고 싶은 안심

### 페르소나 구성 요소

| 항목     | 내용                        |
| ------ | ------------------------- |
| 이름     | 가상의 대표 사용자                |
| 역할     | 작업자, 고령자, 보호자, 관리자, 경비원 등 |
| 상황     | 어떤 환경에서 어떤 위험에 노출되는가      |
| 목표     | 무엇을 안전하게 하고 싶은가           |
| 두려움    | 무엇을 불안해하는가                |
| 현재 대응  | 지금은 어떻게 대처하는가             |
| 한계     | 현재 방식이 왜 부족한가             |
| 서비스 기대 | 어떤 도움을 원하고 있는가            |
| 안심 조건  | 무엇이 있어야 서비스를 신뢰하는가        |

### 예시 페르소나 유형

산업현장:

* 위험행동에 노출되는 작업자
* 여러 현장을 관리하는 EHS 관리자
* 생산성과 안전 사이에서 고민하는 현장 책임자
* AI 감시에 불안감을 느끼는 작업자

돌봄:

* 독거 고령자
* 멀리 사는 보호자
* 방문요양사
* 응급대응 담당자

주거·건물 보안:

* 입주민
* 경비원
* 관리소장
* 외부 방문자
* 관리비 부담자

## 7. 5단계: 위험상태 전이 분석

### 목적

사고를 단일 사건으로 보지 않고, 정상 상태에서 위험 상태로 전이되는 과정으로 분석한다.

### 기본 전이 구조

```text
정상 상태
→ 주의 상태
→ 위험 상태
→ 사고 상태
→ 회복 상태
```

### 분석 항목

| 단계 | 상태 설명           | 관찰 가능한 신호          | 개입 기회   | 필요한 조치 |
| -- | --------------- | ------------------ | ------- | ------ |
| 정상 | 평상시 안전한 상태      | 일반 패턴              | 안전상태 유지 | 모니터링   |
| 주의 | 위험 가능성이 높아지는 상태 | 이상 패턴, 미준수, 불안정 행동 | 조기 경고   | 약한 개입  |
| 위험 | 즉시 개입이 필요한 상태   | 위험구역 진입, 장시간 무응답 등 | 즉시 알림   | 현장 조치  |
| 사고 | 피해가 발생한 상태      | 낙상, 충돌, 침입, 설비 사고  | 구조·대응   | 응급조치   |
| 회복 | 조치 후 복귀 상태      | 조치 완료, 안전 확인       | 재발 방지   | 피드백·개선 |

### 설계 원칙

좋은 안전·안심 서비스는 사고 상태만 감지하지 않는다.
정상에서 주의, 주의에서 위험으로 전이되는 조기 신호를 포착해야 한다.

## 8. 6단계: 사용자 여정지도 작성

### 목적

사용자가 위험상황을 경험하는 전체 흐름을 시간순으로 정리하고, 각 단계에서 불안과 개입 기회를 찾는다.

### 사용자 여정지도 기본 구조

| 단계 | 사용자 행동 | 위험/불안 | 현재 대응 | Pain Point | 서비스 기회 |
| -- | ------ | ----- | ----- | ---------- | ------ |

### 산업현장 예시

| 단계    | 사용자 행동         | 위험/불안        | 현재 대응        | Pain Point | 서비스 기회      |
| ----- | -------------- | ------------ | ------------ | ---------- | ----------- |
| 작업 시작 | 작업자가 현장에 진입    | 위험구역 인지 부족   | 안전교육         | 교육 기억 의존   | 위치 기반 위험 안내 |
| 작업 중  | 장비와 같은 공간에서 작업 | 지게차 접근 위험    | 관리자 순찰       | 실시간 감지 어려움 | AI 위험 감지    |
| 위험 발생 | 작업자가 장비 동선에 접근 | 충돌 가능성       | 경고음 또는 육안 확인 | 늦은 대응      | 현장 알림       |
| 조치    | 작업자가 회피        | 실제 회피 여부 불확실 | 관리자 확인       | 결과 확인 어려움  | 조치 확인       |
| 사후    | near miss 기록   | 반복 위험 파악 어려움 | 수기 보고        | 누락 많음      | 자동 기록·분석    |

## 9. 7단계: Pain Point와 Opportunity 도출

### 목적

사용자 여정에서 발견한 문제를 서비스 기회로 전환한다.

### Pain Point 유형

| 유형     | 설명                          |
| ------ | --------------------------- |
| 감지 공백  | 위험이 발생해도 보이지 않음             |
| 대응 지연  | 누가 조치해야 하는지 늦게 파악           |
| 책임 불명확 | 알림 이후 누가 책임지는지 모호함          |
| 정보 단절  | 감지 결과가 업무 시스템과 연결되지 않음      |
| 불신     | 사용자가 AI를 감시로 느끼거나 알림을 믿지 않음 |
| 확인 부재  | 조치가 완료되었는지 확인하지 못함          |
| 학습 부재  | 반복 위험이 개선으로 이어지지 않음         |

### Opportunity 표현 방식

```text
[문제] 때문에 [사용자]는 [불편/불안/위험]을 경험한다.
따라서 서비스는 [기회]를 제공해야 한다.
```

예시:

```text
CCTV가 사고 후 확인용으로만 사용되기 때문에 관리자는 사고 전 위험을 알기 어렵다.
따라서 서비스는 기존 CCTV를 사고 전 위험 신호를 감지하는 인프라로 전환해야 한다.
```

## 10. 8단계: HMW 질문 도출

### 목적

Pain Point를 해결 가능한 설계 질문으로 바꾼다.

HMW는 “How Might We”의 약자로, “어떻게 하면 ~할 수 있을까?” 형식의 질문이다.

### 좋은 HMW의 조건

* 너무 넓지 않아야 한다.
* 특정 솔루션을 미리 정하지 않아야 한다.
* 사용자와 문제상황이 분명해야 한다.
* 서비스 설계 방향을 열어 주어야 한다.

### 안전·안심 서비스 HMW 예시

```text
HMW 1.
어떻게 하면 사용자가 위험상태로 전이되기 전에 이를 조기에 감지할 수 있을까?

HMW 2.
어떻게 하면 감지된 위험이 단순 알림에 머무르지 않고 실제 조치와 결과 확인으로 이어지게 할 수 있을까?

HMW 3.
어떻게 하면 사용자가 AI를 감시가 아니라 자신의 안전을 돕는 서비스로 받아들이게 할 수 있을까?

HMW 4.
어떻게 하면 관리자가 제한된 시간과 자원을 가장 위험한 상황에 우선 배분할 수 있을까?

HMW 5.
어떻게 하면 반복되는 위험 패턴을 교육, 동선 개선, 설비 개선으로 연결할 수 있을까?
```

## 11. 9단계: 서비스 컨셉 도출

### 목적

HMW 질문을 바탕으로 서비스 방향을 구체화한다.

### 서비스 컨셉 구성 요소

| 항목     | 내용                                                         |
| ------ | ---------------------------------------------------------- |
| 서비스명   | 서비스의 핵심 가치를 담은 이름                                          |
| 핵심 사용자 | 누구를 위한 서비스인가                                               |
| 해결 문제  | 어떤 위험과 불안을 해결하는가                                           |
| 핵심 가치  | 안전 측면 가치와 안심 측면 가치                                         |
| 주요 기능  | 감지, 알림, 조치, 확인, 학습 기능                                      |
| 차별점    | 기존 방식과 다른 점                                                |
| 사례 연결  | Intenseye, Protex AI, Shawmut, Mind Foundry 중 어떤 교훈을 반영하는가 |

### 서비스 컨셉 문장 구조

```text
[서비스명]은 [핵심 사용자]가 [위험상황]으로 전이되기 전에
AI가 위험 신호를 감지하고, 적절한 사람에게 알리며,
조치와 결과 확인을 통해 실제 위험을 줄이고
사용자가 안전하다고 느끼게 하는 [서비스 유형]이다.
```

## 12. 10단계: 폐루프 서비스 구조 설계

### 목적

서비스가 감지에서 끝나지 않고 조치와 결과 확인으로 이어지도록 설계한다.

### 폐루프 구조

```text
위험 신호 수집
→ 위험 감지
→ 위험 해석
→ 사용자별 알림
→ 조치 실행
→ 결과 확인
→ 학습과 개선
→ 안심 피드백
```

### 설계 항목

| 단계       | 설계 내용                     |
| -------- | ------------------------- |
| 위험 신호 수집 | 어떤 데이터를 수집할 것인가           |
| 위험 감지    | 어떤 위험을 AI가 감지할 것인가        |
| 위험 해석    | 위험도와 우선순위를 어떻게 판단할 것인가    |
| 사용자별 알림  | 누구에게 어떤 방식으로 알릴 것인가       |
| 조치 실행    | 누가 어떤 조치를 할 것인가           |
| 결과 확인    | 위험이 해소되었는지 어떻게 확인할 것인가    |
| 학습과 개선   | 반복 위험과 오탐·미탐을 어떻게 반영할 것인가 |
| 안심 피드백   | 사용자에게 안전상태를 어떻게 알려줄 것인가   |

---

## 13. 11단계: AI·데이터 구조 설계

### 목적

서비스가 어떤 데이터를 사용하고 AI가 무엇을 판단하는지 정의한다.

### 데이터 유형

| 데이터 유형  | 예시                          |
| ------- | --------------------------- |
| 영상 데이터  | CCTV, 열화상, 엣지 카메라           |
| 센서 데이터  | 위치, 출입, 움직임, 온도, 가스, 소음     |
| 업무 데이터  | 작업일정, 근무조, 공정, 협력업체, 교육 이력  |
| 환경 데이터  | 날씨, 조도, 혼잡도, 공간 구조          |
| 이력 데이터  | 사고, near miss, 조치 기록, 오탐·미탐 |
| 사용자 피드백 | 알림 확인, 불안도, 서비스 신뢰도         |

### AI 판단 항목

| 판단 항목  | 설명               |
| ------ | ---------------- |
| 감지     | 위험행동 또는 위험상태를 찾음 |
| 분류     | 위험 유형을 구분        |
| 위험도 평가 | 위험 수준을 점수화       |
| 우선순위화  | 먼저 조치해야 할 상황을 추천 |
| 설명     | 왜 위험한지 이유 제시     |
| 학습     | 현장 피드백과 결과를 반영   |

---

## 14. 12단계: 안전층과 안심층 설계

### 목적

실제 위험을 줄이는 기능과 사용자가 신뢰하게 만드는 경험을 분리해서 설계한다.

### 안전층

안전층은 실제 위험을 줄이는 기능이다.

예시:

* 위험행동 감지
* 낙상 감지
* 침입 감지
* 위험구역 진입 감지
* 현장 경고
* 관리자 알림
* 조치 요청
* 조치 완료 확인

### 안심층

안심층은 사용자가 위험이 관리되고 있다고 느끼게 만드는 기능이다.

예시:

* 알림 이유 설명
* 개인정보 보호
* 얼굴·신원 익명화
* 조치 완료 알림
* 보호자 안심 메시지
* 오탐 피드백
* 사용자 통제권
* 감시가 아니라 보호 목적이라는 명확한 커뮤니케이션

---

## 15. 13단계: 운영모델과 비즈니스 모델 설계

### 목적

서비스가 실제 조직에서 지속적으로 운영될 수 있도록 한다.

### 운영모델 설계 항목

| 항목     | 질문                    |
| ------ | --------------------- |
| 운영 주체  | 누가 서비스를 운영하는가         |
| 조치 책임  | 알림을 받으면 누가 행동하는가      |
| 에스컬레이션 | 조치가 되지 않으면 누구에게 넘어가는가 |
| 유지보수   | 모델과 장비는 누가 관리하는가      |
| 피드백    | 오탐·미탐은 어떻게 반영하는가      |
| 보고     | 결과는 누구에게 보고되는가        |

### 비즈니스 모델 설계 항목

| 항목    | 질문                          |
| ----- | --------------------------- |
| 비용부담자 | 누가 비용을 지불하는가                |
| 구매 이유 | 왜 이 서비스에 비용을 지불하는가          |
| 가격 단위 | 카메라당, 공간당, 사용자당, 월 구독형 등    |
| 기대 효과 | 사고 감소, 관리비 절감, 보험, 규제 대응 등  |
| 도입 장벽 | 개인정보, 노조, 비용, 시스템 연동, 운영 부담 |

---

## 16. 14단계: PRD 작성

### 목적

서비스 컨셉을 실제 제품 요구사항으로 구체화한다.

### PRD 기본 항목

| 항목       | 내용                   |
| -------- | -------------------- |
| 제품명      | 서비스 이름               |
| 배경       | 왜 이 서비스가 필요한가        |
| 문제정의     | 해결할 핵심 문제            |
| 목표 사용자   | 주요 사용자와 이해관계자        |
| 핵심 가치    | 안전 가치와 안심 가치         |
| 주요 기능    | 기능 요구사항              |
| 데이터 요구사항 | 필요한 데이터와 수집 방식       |
| AI 판단    | AI가 감지·분류·예측할 내용     |
| 알림 정책    | 누구에게 언제 어떻게 알릴 것인가   |
| 조치 흐름    | 알림 이후 어떤 조치가 이어지는가   |
| 결과 확인    | 조치 완료를 어떻게 확인하는가     |
| KPI      | 성과 측정 지표             |
| 개인정보·윤리  | 감시 우려, 데이터 보호, 책임 구조 |
| MVP 범위   | 초기 구현 범위             |
| 향후 확장    | 추가 기능과 연동 방향         |

---

## 17. 15단계: 평가와 개선

### 목적

설계된 서비스가 실제 안전·안심 가치를 만드는지 평가한다.

### 평가 질문

| 평가 영역   | 질문                       |
| ------- | ------------------------ |
| 문제 적합성  | 실제 중요한 위험을 다루고 있는가       |
| 사용자 가치  | 핵심 사용자가 체감할 가치가 있는가      |
| 안전 효과   | 실제 위험을 줄일 수 있는가          |
| 안심 효과   | 사용자가 신뢰하고 수용할 수 있는가      |
| 운영 가능성  | 누가 조치하고 운영할 수 있는가        |
| 데이터 가능성 | 필요한 데이터를 현실적으로 확보할 수 있는가 |
| 기술 가능성  | AI가 충분히 판단할 수 있는 문제인가    |
| 윤리성     | 감시, 개인정보, 책임 문제가 관리되는가   |
| 비즈니스성   | 비용부담자가 지불할 이유가 있는가       |
| 확장성     | 다른 현장이나 도메인으로 확장 가능한가    |

---

## 18. 단계별 사용 가이드

사용자의 요청에 따라 프로세스를 다르게 적용한다.

### 사용자가 “서비스 설계를 시작하자”고 할 때

1. 문제 이해
2. 문제 재정의
3. 이해관계자 분석
4. 위험상태 전이
5. HMW 도출

### 사용자가 “페르소나를 적용해 달라”고 할 때

1. 핵심 사용자 정의
2. 페르소나 3~4종 작성
3. 페르소나별 Pain Point
4. 페르소나별 서비스 기대
5. 사용자 여정지도 작성

### 사용자가 “PRD로 정리해 달라”고 할 때

1. 서비스 컨셉 정리
2. 기능 요구사항
3. 데이터 요구사항
4. AI 판단 구조
5. 알림·조치·확인 흐름
6. KPI
7. 윤리·운영 이슈

### 사용자가 “강의자료로 정리해 달라”고 할 때

1. 핵심 메시지
2. 도식화 가능한 구조
3. 사례 연결
4. 슬라이드 제목
5. 발표자 노트
6. 토론 질문

---

## 19. 최종 원칙

서비스디자인 프로세스는 단순히 예쁜 산출물을 만드는 과정이 아니다.
안전·안심 서비스에서는 다음 질문에 답하기 위한 과정이다.

* 누구의 안전을 지키는가?
* 무엇이 위험인가?
* 위험은 어떻게 전이되는가?
* 누가 조치할 수 있는가?
* 조치가 끝났는지 어떻게 확인하는가?
* 사용자는 이 서비스를 신뢰할 수 있는가?
* AI는 사람의 판단과 행동을 어떻게 증강하는가?
* 서비스는 지속적으로 학습하고 개선되는가?

나만의 GPT는 이 문서를 참고하여 사용자의 요청을 항상 서비스디자인 프로세스에 따라 구조화해야 한다.
